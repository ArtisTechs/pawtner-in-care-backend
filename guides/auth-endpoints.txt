AUTH ENDPOINTS

Base URL:
http://localhost:8080/api/auth

1. Sign Up
POST /signup
Content-Type: application/json

Important:
- Before calling `/signup`, you must complete OTP verification with purpose `signup`
- `/signup` only works after the latest signup OTP has already been verified
- `/signup` consumes the verified signup OTP during account creation
- `/signup` will fail if the email is already registered

Recommended flow:
1. Call `POST /send-otp` with purpose `signup`
2. Check the email inbox for the 6-digit OTP
3. Call `POST /confirm-otp` with the email, OTP, and purpose `signup`
4. Call `POST /signup` with the same email and account details

Sample body:
{
  "firstName": "Jane",
  "middleName": "Marie",
  "lastName": "Doe",
  "email": "jane@example.com",
  "profilePicture": "https://example.com/profiles/jane.jpg",
  "password": "Test1234!"
}

Notes:
- Role defaults to USER
- USER signups default to active
- User ID is auto-generated as UUID
- Email must be unique
- The signup email must match the verified signup OTP email
- profilePicture is optional and stored as a string
- createdDate is auto-generated on create
- updatedDate is auto-generated on create and update

2. Login
POST /login
Content-Type: application/json

Sample body:
{
  "email": "jane@example.com",
  "password": "Test1234!"
}

Sample success response:
{
  "tokenType": "Bearer",
  "accessToken": "your-token-here",
  "user": {
    "id": "generated-uuid",
    "firstName": "Jane",
    "middleName": "Marie",
    "lastName": "Doe",
    "email": "jane@example.com",
    "profilePicture": "https://example.com/profiles/jane.jpg",
    "role": "USER",
    "active": true,
    "createdDate": "2026-04-04T00:30:00",
    "updatedDate": "2026-04-04T00:30:00"
  }
}

Login note:
- Inactive users are not allowed to log in
- Existing legacy users are marked active on startup by SQL seed data

Main admin seed note:
- One main admin account can be auto-created on startup from env values
- The seeded main admin is created as `ADMIN` and forced active

3. Send OTP
POST /send-otp
Content-Type: application/json

Authorization:
No bearer token required

Sample body:
{
  "email": "jane@example.com",
  "purpose": "signup"
}

Allowed purpose values:
- signup
- login
- reset-password

Signup OTP note:
- If purpose is `signup`, the request will fail when the email is already registered

Forgot password OTP note:
- If purpose is `reset-password`, the request will fail when the email is not registered

4. Confirm OTP
POST /confirm-otp
Content-Type: application/json

Authorization:
No bearer token required

Sample body:
{
  "email": "jane@example.com",
  "otp": "123456",
  "purpose": "signup"
}

Notes:
- For purpose `signup`, confirming the OTP marks it as verified
- The verified signup OTP is consumed later by `/signup`
- For purpose `reset-password`, confirming the OTP marks it as verified
- The verified reset-password OTP is consumed later by `/reset-password`

5. Reset Password
POST /reset-password
Content-Type: application/json

Authorization:
No bearer token required

Important:
- Use purpose `reset-password` when requesting the OTP
- You must call `/confirm-otp` first with purpose `reset-password`
- `/reset-password` only works after the latest reset-password OTP has already been verified
- `/reset-password` must include the same 6-digit OTP from the email
- `/reset-password` consumes the verified OTP after the password is changed

Recommended flow:
1. Call `POST /send-otp` with purpose `reset-password`
2. Check the email inbox for the 6-digit OTP
3. Call `POST /confirm-otp` with the email, OTP, and purpose `reset-password`
4. Call `POST /reset-password` with the email, OTP, and new password

Sample body:
{
  "email": "jane@example.com",
  "otp": "123456",
  "newPassword": "NewPass123!",
  "confirmPassword": "NewPass123!"
}

Sample success response:
{
  "email": "jane@example.com",
  "message": "Password reset successfully"
}
